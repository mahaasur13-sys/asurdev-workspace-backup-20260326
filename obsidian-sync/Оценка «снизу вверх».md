# Bottom-Up Estimation (Оценка «снизу вверх»)

## Краткое определение

**Bottom-Up Estimation** — это метод оценки проекта, при котором общая стоимость, сроки и трудозатраты определяются путем **суммирования оценок отдельных, максимально мелких задач** проекта. Это подход «от частного к общему».

---

## Когда применяется?

*   **На этапе детального планирования**, когда утверждено описание содержания проекта (Scope) и создана **Иерархическая структура работ (ИСР/WBS)**.
*   **Требуется высокая точность** (например, для фиксированных бюджетов, жестких дедлайнов).
*   **Проекты малого и среднего масштаба**, где количество задач позволяет провести детальную оценку без чрезмерных затрат времени.
*   **Критически важна приверженность команды**, так как исполнители сами участвуют в оценке.

---

## Пошаговый алгоритм

### Шаг 1: Декомпозиция проекта (Создание WBS)
Проект разбивается на иерархическую структуру работ (WBS) до уровня **пакетов работ** или даже **конкретных задач**, которые можно оценить с высокой точностью (идеально — от 8 до 80 часов).

### Шаг 2: Назначение ответственных
Каждая задача назначается члену команды или эксперту, который будет ее выполнять или обладает наибольшей экспертизой в этой области.

### Шаг 3: Детальная оценка задач
*   **Кто оценивает:** Непосредственные исполнители.
*   **Что оценивается:**
    *   **Трудозатраты** (в человеко-часах или днях).
    *   **Продолжительность** (в календарных днях с учетом зависимостей и ресурсов).
    *   **Стоимость** (зарплата, материалы, оборудование).
*   **Методы оценки задач:** Могут использоваться экспертные оценки, аналогии, трехточечная оценка (PERT).

### Шаг 4: Агрегация (Суммирование)
Оцен
### Шаг 5: Учет рисков и буферизация
К итоговой сумме добавляется **резерв на непредвиденные обстоятельства** (буфер, обычно 10-20%), чтобы учесть неопределенность и риски.

---

## Преимущества и недостатки

| Преимущества (+) | Недостатки (–) |
| :--- | :--- |
| ✅ **Высокая точность** (оценка от исполнителей) | ❌ **Трудоемкость и дороговизна** процесса |
| ✅ **Принятие и ответственность команды** | ❌ **Требует четко определенного объема (WBS)** |
| ✅ **Глубокое понимание проекта** менеджером | ❌ **Медлительность** (не для быстрых решений) |
| ✅ **Надежная база для контроля выполнения** | ❌ **Риск "накопления погрешностей"** в крупных проектах |

---

## Сравнение с Top-Down Estimation

| Критерий | Bottom-Up (Снизу вверх) | Top-Down (Сверху вниз) |
| :--- | :--- | :--- |
| **Точность** | Высокая | Низкая / Средняя |
| **Скорость оценки** | Медленная | **Быстрая** |
| **Затраты на оценку** | Высокие | **Низкие** |
| **Участие команды** | **Активное** | Минимальное (менеджмент) |
| **Стадия применения** | Детальное планирование | Инициирование, пре-сейл |
| **Основа** | Детальная WBS | Аналоги, экспертиза, параметры |

## Практический пример

**Проект:** Запуск нового лендинга.

1.  **WBS:**
    *   1.1. Анализ (1.1.1 - Анализ конкурентов, 1.1.2 - Составление ТЗ)
    *   1.2. Дизайн (1.2.1 - Прототип, 1.2.2 - Визуальный дизайн)
    *   1.3. Разработка (1.3.1 - Верстка, 1.3.2 - Программирование)
    *   1.4. Тестирование

2.  **Оценка задач (в часах):**
    *   1.1.1 = 8ч, 1.1.2 = 6ч → **Пакет "1.1. Анализ" = 14ч**
    *   1.2.1 = 12ч, 1.2.2 = 16ч → **Пакет "1.2. Дизайн" = 28ч**
    *   1.3.1 = 24ч, 1.3.2 = 32ч → **Пакет "1.3. Разработка" = 56ч**
    *   1.4.1 = 10ч → **Пакет "1.4. Тестирование" = 10ч**

3.  **Агрегация:**
    *   Общая оценка трудозатрат = 14ч + 28ч + 56ч + 10ч = **108 часов**.

4.  **Буфер:**
    *   Добавляем резерв 15%: 108 * 0.15 ≈ 16ч.
    *   **Итоговая оценка: 124 часа.**

## Ключевой вывод

**Bottom-Up Estimation** — это инструмент для проектов, где **точность и обоснованность плана критически важны**. Он требует значительных усилий на старте, но эти инвестиции окупаются за счет снижения рисков перерасхода бюджета и срыва сроков на поздних стадиях проекта. Используйте его, когда у вас есть время, команда и четкое видение объема работ.ки суммируются снизу вверх по иерархии WBS:
